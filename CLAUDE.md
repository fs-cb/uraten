# CLAUDE.md — URATEN サイト実装の前提と制約

このリポジトリは URATEN（福岡・天神のサブカルクリエイター向け24時間Webラジオ）の公開サイト。
本ファイルは実装時に必ず守る決定事項の正本。ここに書かれた制約と衝突する実装・提案はしない。
**設計判断（アーキテクチャの変更・スキーマの変更・新しい仕組みの追加）はこのファイルの範囲外であり、実装者が独断で行わない。提案として運営者に返すこと。**

---

## 1. アーキテクチャの不変条件（違反禁止）

1. **Sheets は審査台、JSON は公開データ。両者を直結させない。**
   - Google Sheets には未審査の申請・運営メモが含まれる。ブラウザ（公開サイト）から Sheets を直接読む実装（「ウェブに公開」CSV / gviz / API直叩き等）は、どんなに効率的でも**絶対に採用しない**。
   - 公開サイトが読んでよいデータは、リポジトリ内の静的 JSON（`data/*.json`）のみ。
2. **状態（DB・サーバーサイド処理）を持たない。**
   - 予約状況・残数・会員などの状態管理を導入しない。サイトは純粋な静的サイト（Cloudflare Pages・ビルドコマンドなし）。
   - 「便利になるから」を理由に DB / KV / Workers / サーバーレス関数を足す提案をしない。導入は運用が限界に来たと運営者が判断した時のみ。
3. **公開のトリガーは手動。**
   - Sheets→JSON の変換・公開は、運営者が承認後に手で1コマンド実行する（承認行為と公開行為の統合）。
   - 書き込み検知や cron による自動公開は実装しない（未審査データが公開に流れる事故、および承認と公開の分離を防ぐため）。
   - push 前に git diff で生成 JSON を目視確認できる形を保つ。
4. **git push が公開。** push 後は Cloudflare Pages が自動デプロイする。デプロイ設定（フレームワークなし・ビルドなし・出力ルート）は変更しない。

## 2. セキュリティ（違反禁止）

- **service account の鍵 JSON をリポジトリに絶対に入れない。** 鍵はリポジトリ外に置き、パスは環境変数か設定ファイル（gitignore 対象）で渡す。`.gitignore` に鍵ファイルのパターンを必ず入れる。
- **公開 JSON（`data/*.json`）に運営用情報を出さない。** 審査メモ・主催者の連絡先・誓約チェックの生データ・Tally の内部 ID 等は Sheets に留める。公開 JSON に載せてよいのは公開表示に必要な項目のみ。
- 公開サイトに申請者が直接書き込める経路を作らない。R2 / リポジトリへの書き込みは運営（変換スクリプト）のみ。
- 申請由来の文字列（イベント名・会場名等）は DOM へ `textContent` で挿入する。`innerHTML` への直接展開をしない（既存 `js/app.js` の方式を維持）。

## 3. リポジトリ構成

```
index.html        … サイト本体（HTML構造のみ）
css/style.css     … 全スタイル
js/app.js         … 全スクリプト（イベントカレンダー描画 renderCal を含む）
data/events.json  … 公開イベントデータ（現状サンプル。将来は変換スクリプトの出力先）
README.md         … デプロイ手順・スキーマ定義
CLAUDE.md         … 本ファイル
```

- これ以上のファイル分割（CSS複数化・JSモジュール化・ビルドツール導入・フレームワーク導入）は行わない。
- 相対パス前提（`./css/` `./js/` `./data/`）。階層構造を変えない。

## 4. events.json スキーマ（連携の契約・勝手に変えない）

`js/app.js` の描画ロジックと、将来の Sheets→JSON 変換スクリプトの両方がこのスキーマに依存する。
フィールド名・構造の変更は両者同時の変更を要するため、独断で変更しない。

```json
{
  "generated_at": "ISO 8601日時",
  "events": [
    {
      "id":    "一意なID（形式自由）",
      "name":  "イベント名",
      "date":  "YYYY-MM-DD（必須）",
      "time":  "HH:MM（任意）",
      "venue": "会場名（任意）",
      "area":  "エリア（任意）",
      "url":   "告知ページURL（任意。あれば行全体がリンク）"
    }
  ]
}
```

- フロント側仕様：過去日は非表示・日付昇順・読み込み中/0件/エラーの3状態表示あり。
- 承認フラグは Sheets 側で**申請単位**（イベント単位ではない）。変換スクリプトは承認済み申請の行を最大10イベントに展開して JSON に吐く。

## 5. データパイプライン（全体像）

```
申請者 → Tally フォーム（ログイン不要）
  → Google Sheets（種別ごとにシート＝審査台。Tally 連携で自動追記）
  → 運営が審査し、手動列「承認」を立てる
  → 変換スクリプト（手動実行）: 承認済み行のみ抽出 → data/events.json 生成 → git push
  → Cloudflare Pages 自動デプロイ → 公開
```

- Sheets の列構成の正本は **Tally フォームの質問項目**。Sheets を手で先に作らない。
- 運営用の手動列（承認・審査メモ・掲載終了日）は Tally 生成列の右側に追加し、Tally は触らない。
- **変換スクリプトは Tally フォーム作成 → Sheet 生成 → 列ヘッダ確定の後に実装する。** 列構成を想像で仮定してスクリプトを先に書かない。

## 6. ラジオプレーヤー実装の必須要件（実装時）

プレーヤーは本サービスで最も故障しやすく、かつ事業の最重要仮説（ながら聴取・回遊）の生命線。
以下を必須要件とし、**素の `<audio>` タグだけの実装は不可**。

1. **Media Session API を実装する。** ロック画面・コントロールセンターに再生 UI とメタデータ（番組名等）を出す。メタデータは AzuraCast の Now Playing API（Websocket / SSE）から取得する。
2. **瞬断からの自動再接続を実装する。** モバイル回線の瞬断でストリームが切れた際、切断検知→リトライで自動復帰する。素の `<audio>` は切れたら止まったままであり、これを放置すると回遊計測（セッション連結）のデータも汚れる。
3. **音のエンジン部分は車輪の再発明をしない。** AzuraCast（オープンソース）の公開プレーヤーの実装（再接続・Media Session・フォーマットフォールバック）を土台・参考にする。自作するのは UI の殻。
4. **バックグラウンド再生の実機検証は未実施**（VPS 未契約のため）。iOS Safari / Android Chrome での「他アプリ移動」「画面ロック」「機内モード数秒→復帰」の挙動確認はゲート C-1（放送技術検証）で行う。検証前に「動く前提」の断定をしない。
